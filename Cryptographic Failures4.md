# Сбои в криптографии (Cryptographic Failures)

## Цель работы
Проанализировать корректность применения криптографических механизмов в веб-приложении, включая защиту передаваемых данных, хранение учетных данных и использование токенов аутентификации. Определить наличие ошибок в реализации шифрования, хэширования или управлении ключами, которые могут привести к раскрытию конфиденциальной информации либо компрометации учетных записей пользователей.

## Ход выполнения

### 1) Разведка
Для анализа была выбрана торговая платформа **UspeshnyiTrader**. При попытке авторизации был перехвачен HTTP-запрос.

**Выявлено:**
- Пароль передаётся в открытом виде внутри тела POST-запроса (`password: 123456`)
- Передача осуществляется по протоколу HTTPS, что обеспечивает защиту канала связи на уровне TLS

### 2) Атака (SSLstrip — понижение HTTPS до HTTP)
Для демонстрации криптографического сбоя, связанного с полаганием только на TLS, была проведена атака **SSLstrip**. Суть атаки заключается в принудительном понижении защищённого соединения HTTPS до незащищённого HTTP, что позволяет перехватывать учётные данные и сессионные куки.

**Топология сети:**
- Атакующий (Kali Linux) — `192.168.0.12`
- Жертва (Windows) — `192.168.0.11`
- Шлюз — `192.168.0.1`

### 3) Эксплуатация
После запуска атаки жертва (пользователь `trader1`) заходит на сайт по адресу `http://192.168.0.11:5000` и выполняет авторизацию. Вся передаваемая информация проходит через атакующего.

**Перехваченные данные в логах sslstrip:**
![Логи sslstrip с кукой](https://github.com/user-attachments/assets/afc31f95-26e7-4371-85b2-1986f649e117)

**В логах были зафиксированы:**
- Метод `GET /Trading` с сессионной кукой `UspeshnyiTrader.Session`
- Метод `POST /Account/Login` с параметрами `username=trader1&password=123456`

Это доказывает, что злоумышленник, находящийся в одной сети с жертвой, может перехватить учётные данные и сессионную куку в открытом виде, несмотря на то, что приложение использует HTTPS.

## Выводы о защищенности
В ходе анализа выявлены следующие **криптографические сбои** в реализации механизма аутентификации:

| № | Проблема | Риск |
|---|----------|------|
| 1 | Передача пароля в открытом виде в теле запроса | Компрометация учётных данных |
| 2 | Отсутствие защиты от понижения протокола (SSLstrip) | Перехват трафика в локальной сети |
| 3 | Полагание только на TLS для защиты данных | Нарушение принципа эшелонированной защиты |

Приложение использует HTTPS, однако при атаке **SSLstrip** злоумышленник может принудительно понизить соединение до HTTP и перехватить конфиденциальные данные.

Данная атака классифицируется как:
- **CWE-319: Cleartext Transmission of Sensitive Information** — передача пароля в открытом виде
- **CWE-523: Unprotected Transport of Credentials** — незащищённая передача учётных данных

### Рекомендации по устранению
1. Внедрить **HTTP Strict Transport Security (HSTS)** для принудительного использования HTTPS
2. Установить флаги кук `Secure; HttpOnly; SameSite=Strict`
3. Использовать шифрование чувствительных данных на прикладном уровне (например, хеширование пароля на клиенте)
4. Реализовать механизм привязки сессии к IP-адресу или User-Agent
