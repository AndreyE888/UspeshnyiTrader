# Ошибки логирования и сбои мониторинга (Security Logging and Monitoring Failures)

## Цель работы
Проанализировать механизмы логирования и мониторинга в веб-приложении на предмет их эффективности при обнаружении атак. Выявить недостатки в регистрации событий безопасности, отсутствие алертов и реакций на подозрительную активность, а также оценить возможность длительного нахождения злоумышленника в системе без обнаружения.

## Ход выполнения

### 1) Разведка
Для анализа был выбран модуль Command Injection в DVWA. Данный модуль позволяет выполнять системные команды через уязвимый параметр. Для доступа к логам использован запущенный Docker-контейнер с DVWA. Логи Apache внутри контейнера находятся по стандартному пути `/var/log/apache2/`.

**Доступ к логам контейнера и их содержимое до атаки:**
![Скриншот 1: Просмотр логов access.log до атаки](https://github.com/user-attachments/assets/0259a0bc-8d64-4e35-ba7d-81614eec5e83)

В логах на данный момент зафиксированы лишь переходы пользователя по страницам сайта: `login.php`, `setup.php`, `index.php`, `security.php`. Никакой подозрительной активности не зарегистрировано.

### 2) Атака
Для проверки эффективности мониторинга проведена атака Command Injection. В поле ввода IP-адреса отправлен следующий payload:

`127.0.0.1 && cat /etc/passwd`

Сервер успешно выполнил команду и вернул содержимое файла `/etc/passwd`:

![Скриншот 2: Результат атаки (содержимое /etc/passwd)](https://github.com/user-attachments/assets/d43f3fc5-a02a-4bde-9710-94307f3eefca)

### 3) Эксплуатация
Сразу после проведения атаки проанализированы логи Apache.

**Логи access.log после атаки:**
![Скриншот 3: Логи access.log после атаки](https://github.com/user-attachments/assets/3c47a49e-83ca-43de-8d8c-9ec15e5b1355)

В логах зафиксирован факт POST-запроса к `/vulnerabilities/exec/`. Однако **информация о переданных параметрах (payload) отсутствует**. Администратор видит только факт обращения к уязвимому эндпоинту, но не может узнать, какая команда была выполнена (`cat /etc/passwd`).

**Логи error.log:**
![Скриншот 4: Логи error.log (пусто или без атаки)](https://github.com/user-attachments/assets/70464d07-b633-4a81-94d8-ac25c9fb8e20)

В файле `error.log` также отсутствуют записи, указывающие на подозрительную активность. Зафиксирована только стандартная информация о запуске сервера и не связанная с безопасностью ошибка PHP.

### 4) Анализ конфигурации логирования
Apache по умолчанию использует формат логов `common` или `combined`, который **не включает параметры GET/POST-запросов**. Для логирования параметров требуется явно настроить `LogFormat` с директивой `%q`. В данном случае такая настройка отсутствует, что является недостатком конфигурации (Security Misconfiguration).

## Выводы о защищенности
В результате анализа выявлено, что система ведет базовое логирование HTTP-запросов в стандартной конфигурации Apache, однако:

1. **Параметры запроса (payload) не сохраняются в логах** — администратор не может определить, какая команда была выполнена злоумышленником.
2. **Отсутствует мониторинг в реальном времени** — система не генерирует алерты при обнаружении подозрительной активности (например, выполнение системных команд).
3. **Нет уведомлений администратора** — атака может оставаться незамеченной длительное время.

Проведенная атака зафиксирована в логах лишь как обычный POST-запрос без сохранения переданных параметров, что делает невозможным расследование инцидента. В логах ошибок также отсутствуют записи о подозрительной активности, система не блокирует атакующего и не уведомляет администратора.

Данная проблема классифицируется как **A09: Security Logging and Alerting Failures** (OWASP Top 10:2025). В реальном сценарии (например, медицинская система) отсутствие логирования параметров и мониторинга привело бы к тому, что злоумышленник мог бы изменять критически важные данные (диагнозы, дозировки лекарств) без возможности обнаружения в течение многих лет.

**CWE:**
- **CWE-778: Insufficient Logging** — недостаточное логирование (параметры запроса не сохраняются)
- **CWE-223: Omission of Security-relevant Information** — пропуск важной информации безопасности

### Рекомендации по устранению
1. **Настроить логирование параметров запросов** — изменить `LogFormat` в конфигурации Apache, добавив `%q` для логирования строки запроса.
2. **Внедрить SIEM-систему** для централизованного сбора и анализа логов.
3. **Настроить алерты на подозрительные паттерны** (`cat /etc/passwd`, `whoami`, `&&`, `|`, `;`).
4. **Использовать WAF (Web Application Firewall)** для блокировки атак в реальном времени.
5. **Регулярно проводить аудит логов** и настраивать уведомления при необычной активности.
