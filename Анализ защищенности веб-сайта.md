# Анализ защищенности веб-приложения UspeshnyiTrader

## Объект испытаний

В качестве объекта испытаний рассматривается веб-приложение **UspeshnyiTrader** — торговая платформа для бинарных опционов. Пользователи могут просматривать котировки, совершать сделки, управлять балансом и редактировать профиль. Администратор имеет доступ к расширенным функциям управления.

Серверная часть реализована на **ASP.NET Core** с использованием фреймворка **Entity Framework Core**.  
Для хранения данных применяется **СУБД PostgreSQL**.

В приложении реализованы следующие основные страницы:
- **/Account/Login** и **/Account/Register** — страницы аутентификации и регистрации;
- **/Trading** — основная торговая страница с котировками и сделками;
- **/Account/Profile** — профиль пользователя;
- **/Admin** — административная панель (доступна только администратору).

Функциональность системы включает регистрацию и аутентификацию пользователей, просмотр котировок в реальном времени, совершение сделок (открытие позиций), изменение баланса, а также разграничение доступа на основе сессионных кук. Пользователи с ролью администратора имеют доступ к управлению пользователями и просмотру статистики. Контроль доступа реализован с использованием сессионных кук (`UspeshnyiTrader.Session`), передаваемых в заголовках HTTP-запросов.

Хранение данных организовано в реляционной базе данных, включающей следующие основные сущности:
- **Users** — содержит логин, хеш пароля, открытый пароль (PlainPassword), баланс, дату создания;
- **Instruments** — содержит торговые инструменты (валюты, акции);
- **Trades** — содержит информацию о сделках: тип, объём, цена открытия, статус;
- **UserBalances** — история изменения баланса пользователей.

Развертывание веб-приложения выполнено на локальном сервере под управлением операционной системы **Windows** с использованием **Kestrel** (встроенный веб-сервер ASP.NET Core).

## Цель проведения испытаний

Оценка уровня защищенности веб-приложения и выявление потенциальных уязвимостей, влияющих на конфиденциальность, целостность и доступность данных.

## Общие положения

#### Основания для проведения испытаний

Основанием для проведения испытаний является лабораторная работа **«Анализ защищенности веб-ресурса»** по курсу «Информационная безопасность web-ресурсов».

#### Продолжительность, время и место испытаний

Продолжительность, время и место проведения испытаний определяются самостоятельно. Испытания проводятся в локальной среде с использованием персонального компьютера.

#### Участвующие стороны

Испытания проводятся одним лицом — студентом, выполняющим данную лабораторную работу.

#### Нормативно-правовая база

При проведении анализа защищенности учитываются рекомендации и методологии тестирования веб-приложений, включая подходы, описанные в:

**OWASP Top 10** — перечень наиболее критичных уязвимостей веб-приложений.

**OWASP Application Security Verification Standard (ASVS)** — стандарт требований к безопасности.

**OWASP Web Security Testing Guide (WSTG)** — руководство по тестированию безопасности.

## Объем испытаний

В рамках проведения испытаний выполняется анализ защищенности веб-приложения с целью выявления уязвимостей, возникающих при обработке пользовательских данных и взаимодействии клиента с сервером.

В процессе тестирования рассматриваются следующие направления:
- анализ конфигурации веб-сервера и используемых технологий;
- анализ механизмов регистрации и аутентификации пользователей;
- проверка корректности разграничения прав доступа (IDOR, эскалация привилегий);
- исследование обработки входных данных, передаваемых с клиента;
- анализ защищенности API и серверных маршрутов;
- проверка устойчивости к типовым атакам на веб-приложения (брутфорс, логирование).

## Вывод

По результатам проведённого анализа защищённости веб-приложения **UspeshnyiTrader** установлено, что приложение содержит как удовлетворительные механизмы безопасности, так и ряд критических уязвимостей.

### Положительные аспекты:
- Используется HTTPS для передачи данных.
- Применяются CSRF-токены на формах авторизации.
- Сессионные куки защищены флагами `HttpOnly` и `Secure`.
- SQL-инъекции невозможны благодаря параметризованным запросам Entity Framework Core.

### Выявленные недостатки:

| № | Проблема | Риск |
|---|----------|------|
| 1 | **IDOR (Insecure Direct Object Reference)** — доступ к профилям других пользователей через изменение ID в URL | Получение чужих персональных данных и эскалация привилегий |
| 2 | **Вертикальная эскалация привилегий** — обычный пользователь может получить данные администратора | Компрометация учётной записи администратора |
| 3 | **Отсутствие блокировки при брутфорсе** | Возможность подбора паролей методом перебора |
| 4 | **Пароль передаётся в открытом виде** (хоть и внутри TLS) | Риск перехвата при компрометации TLS или сниффинге |
| 5 | **Логирование чувствительных данных** (паролей) в консоль | Утечка учётных данных при доступе к логам |

### Рекомендации по устранению:

1. **Устранить IDOR** — добавить проверку прав доступа на серверной стороне при запросе профиля по ID.
2. **Внедрить механизм блокировки учётной записи** после нескольких неудачных попыток входа.
3. **Хранить пароли только в хешированном виде** — удалить поле `PlainPassword` из базы данных.
4. **Не логировать чувствительные данные** — убрать вывод паролей в консоль.
5. **Внедрить многофакторную аутентификацию** для критических операций (изменение баланса, вывод средств).

В целом, веб-приложение требует доработки механизмов контроля доступа и защиты учётных данных для повышения устойчивости к современным угрозам информационной безопасности.
