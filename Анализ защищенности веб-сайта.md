# Анализ защищенности веб-приложения UspeshnyiTrader

## Объект испытаний

В качестве объекта испытаний рассматривается веб-приложение **UspeshnyiTrader** — торговая платформа для бинарных опционов. Пользователи могут просматривать котировки, совершать сделки, управлять балансом и редактировать профиль.

Серверная часть реализована на **ASP.NET Core** с использованием фреймворка **Entity Framework Core**. Для хранения данных применяется **СУБД PostgreSQL**.

В приложении реализованы следующие основные страницы:
- **/Account/Login** и **/Account/Register** — страницы аутентификации и регистрации;
- **/Trading** — основная торговая страница с котировками;
- **/Account/Profile** — профиль пользователя;
- **/Account/Profile/{id}** — уязвимый эндпоинт для просмотра любых профилей.

Функциональность системы включает регистрацию и аутентификацию пользователей, изменение баланса, а также разграничение доступа на основе сессионных кук (`UspeshnyiTrader.Session`). Контроль доступа на некоторых эндпоинтах отсутствует или реализован некорректно.

Развертывание веб-приложения выполнено на локальном сервере под управлением операционной системы **Windows** с использованием **Kestrel**.

## Цель проведения испытаний

Оценка уровня защищенности веб-приложения и выявление потенциальных уязвимостей, влияющих на конфиденциальность, целостность и доступность данных.

## Общие положения

#### Основания для проведения испытаний

Основанием для проведения испытаний является лабораторная работа **«Анализ защищенности веб-ресурса»** по курсу «Информационная безопасность web-ресурсов».

#### Продолжительность, время и место испытаний

Испытания проводятся в локальной среде с использованием персонального компьютера.

#### Участвующие стороны

Испытания проводятся одним лицом — студентом, выполняющим данную лабораторную работу.

#### Нормативно-правовая база

При проведении анализа защищенности учитываются рекомендации и методологии тестирования веб-приложений, включая подходы, описанные в:

**OWASP Top 10** — перечень наиболее критичных уязвимостей веб-приложений.

**OWASP Application Security Verification Standard (ASVS)** — стандарт требований к безопасности.

**OWASP Web Security Testing Guide (WSTG)** — руководство по тестированию безопасности.

## Объем испытаний

В процессе тестирования рассматриваются следующие направления:
- анализ конфигурации веб-сервера и используемых технологий;
- анализ механизмов аутентификации и управления сессиями;
- проверка корректности разграничения прав доступа (IDOR, эскалация привилегий);
- проверка устойчивости к типовым атакам (брутфорс, утечка данных в логах).

## Вывод

По результатам проведённого анализа защищённости веб-приложения **UspeshnyiTrader** выявлены следующие недостатки и их риски:

| № | Проблема | Риск |
|---|----------|------|
| 1 | **IDOR (Insecure Direct Object Reference)** — доступ к данным других пользователей через изменение ID. | Получение чужих персональных данных и паролей. |
| 2 | **Вертикальная эскалация привилегий** — обычный пользователь получает данные администратора. | Полная компрометация административной учётной записи. |
| 3 | **Отсутствие блокировки при брутфорсе**. | Возможность подбора паролей методом перебора. |
| 4 | **Пароль передаётся в открытом виде** внутри тела запроса. | Риск перехвата учётных данных при компрометации канала связи. |
| 5 | **Логирование чувствительных данных** (паролей). | Утечка учётных данных при доступе к логам. |
| 6 | **Отсутствие проверки бизнес-логики** (изменение баланса). | Несанкционированное изменение критически важных данных. |

### Рекомендации по устранению:
1. **Устранить IDOR** — добавить проверку прав доступа на серверной стороне при запросе профиля по ID.
2. **Внедрить механизм блокировки учётной записи** после нескольких неудачных попыток входа.
3. **Хранить пароли только в хешированном виде** и не передавать их в открытом виде в ответах API.
4. **Не логировать чувствительные данные** — убрать вывод паролей и токенов в консоль.
5. **Внедрить валидацию на сервере** для всех критических операций (изменение баланса).
6. **Внедрить многофакторную аутентификацию (MFA)** для административных действий.

В целом, веб-приложение требует доработки механизмов контроля доступа, защиты учётных данных и проверки бизнес-логики для обеспечения приемлемого уровня безопасности.
